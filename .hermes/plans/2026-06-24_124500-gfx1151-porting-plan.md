# gfx1151 (RDNA 3) Porting Plan

> **For Hermes:** Use subagent-driven-development to implement this plan task-by-task.
> **Status:** Planning phase — no code written yet.

**Goal:** Make SGLang compile and run on AMD Radeon RX 7600-series GPUs (gfx1151, RDNA 3 architecture).

**Architecture context:** gfx1151 is a consumer RDNA 3 GPU (Navi 33). It differs from the currently supported CDNA datacenter GPUs (gfx942 MI300, gfx950 MI350) in several key ways:
- Uses **FP8 FNUIZ** format (same as gfx942, different from gfx950's E4M3F)
- Smaller LDS (shared memory) budget than CDNA — typically 64KB per CU vs 128KB+
- **No AITER** support (pre-built ASM binaries are CDNA-only)
- No hardware NVLink/RadeonRDMA (only PCIe P2P)
- Lower VRAM (8-16GB typical vs 192GB on MI300X)
- ROCm Triton **does** support RDNA 3

**Tech Stack:** HIP, ROCm 6.x+, ROCm Triton, PyTorch ROCm, sgl-kernel (AOT + JIT)

def is_gfx11_supported():
    """Returns whether current GPU is gfx11xx (RDNA 3, e.g. gfx1151)."""

The `tvm_ffi` library may support `TVM_FFI_ROCM_ARCH_LIST` env var. We need to investigate whether this env var exists and what value it expects for gfx1151.

```python
@contextmanager
def _jit_compile_context():
    if is_hip_runtime():
        # TVM FFI on ROCm uses TVM_FFI_ROCM_ARCH_LIST to set the target arch.
        # Fall back to the rocm-arch from torch if available.
        env_key = "TVM_FFI_ROCM_ARCH_LIST"
        old_value = os.environ.get(env_key, None)
        try:
            gcn_arch = torch.cuda.get_device_properties(0).gcnArchName
            # Take only the base arch (e.g. "gfx1151" from "gfx1151:...")
            rocm_arch = gcn_arch.split(":")[0]
            if rocm_arch:
                os.environ[env_key] = rocm_arch
            yield
        finally:
            if old_value is None:
                os.environ.pop(env_key, None)
            else:
                os.environ[env_key] = old_value
        return
    # ... existing CUDA path unchanged
```

**Note:** This requires verifying that `tvm_ffi` actually supports `TVM_FFI_ROCM_ARCH_LIST`. If not, JIT kernels may fall back to a generic `--offload-arch=` flag or need `-triton` level dispatch instead. **This is a risk point** — see Risks below.

## Risks, Tradeoffs, and Open Questions

### High Risk: TVM FFI JIT compilation on ROCm
The JIT compilation pipeline uses `tvm_ffi` which is primarily tested on CUDA. The `_jit_compile_context()` has an explicit `# TODO: support ROCm TVM_FFI_ROCM_ARCH_LIST if needed`. If `tvm_ffi` doesn't handle ROCm archs, JIT kernels will either:
1. Fall back to a default architecture (may produce wrong code)
2. Crash at compile time

**Mitigation:** Test a single JIT kernel load on gfx1151 first. If it fails, the JIT kernel path may need a separate workaround (pre-compiling for a generic target or using `--offload-arch=gfx1151`).

### Medium Risk: Quick AllReduce kernel correctness
QAR uses in-register quantized reduction with wave-level primitives. Even though RDNA 3 uses 64-wide wavefronts (same as CDNA), the instruction scheduling and register pressure characteristics differ. The kernel may produce wrong results or deadlock.

**Mitigation:** Env-gated (disabled by default) until validated with correctness tests.

### Low Risk: VRAM constraints
gfx1151 GPUs typically have 8-16GB VRAM. SGLang is designed for large-scale LLM serving. Many models will OOM. This is not a porting issue but a practical limitation — users should stick to 0.5B-3B models with FP16/BF16.

### Open Question: Does `tvm_ffi` support ROCm at all?
The JIT kernel infra depends on `tvm_ffi`. If this library does not support HIP/ROCm, then **all JIT kernels** (which includes many optimized paths: moe, attention, quantization) will be unavailable on gfx1151. The fallback paths would use generic Triton or PyTorch ops.

### Open Question: What ROCm version is needed for gfx1151?
gfx1151 (Navi 33) requires ROCm 6.2+ for full PyTorch support. ROCm 5.x does not support this GPU. The Dockerfile currently targets ROCm 7.0.0 and 7.2.0, which should work. Verify:

```bash
rocminfo | grep "Name:"
```

### Stretch: Triton kernel port
SGLang has many Triton kernels. ROCm Triton works on RDNA 3, so these should compile. But some use CUDA-specific features (`torch.heuristic`, SM-specific tile sizes). These may need tuning for RDNA3's 64-wide warps and different memory hierarchy.
